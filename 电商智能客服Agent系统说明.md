# 电商智能客服 Agent 系统

**项目周期：2025年12月 - 2026年02月**

## 项目简介

针对传统电商客服严重依赖人工、且常规大模型客服容易产生事实幻觉的痛点，主导开发基于 FastAPI 的企业级 Agent 智能客服后端系统。系统围绕用户意图识别、多轮上下文记忆、权限校验、动态工具调用与真实业务数据回填构建完整执行闭环，并结合 Guardrail、安全权限沙箱、Redis 高频接口限流与多级 Fallback 容灾机制，将大模型能力转化为高可用、安全合规的业务级生产力工具。

## 个人工作

### Agent 核心总控与全链路闭环

设计 `SupportAgentOrchestrator` 作为请求生命周期编排层，统一负责用户输入接收、多轮历史上下文提取、意图识别、工具调用决策、真实业务结果回填与最终自然语言生成。该设计将原本被动式的客服问答改造成基于真实业务状态驱动的 Agent 执行链路，使系统不再单纯依赖大模型自由生成，而是通过业务数据约束回答边界。

在执行流程上，系统先从 `support_messages` 中提取最近多轮对话上下文，再通过客制化 Prompt 将用户请求路由到 `order_query`、`policy_query`、`product_inquiry`、`chitchat` 等隔离分支。不同分支采用不同的系统提示词、工具上下文和降级策略，从而避免订单查询、规则咨询、商品咨询和闲聊混在同一 Prompt 中导致回答失焦或事实污染。

### 基于真实场景的 Function Calling 框架

构建面向电商核心业务的工具调用机制，针对订单查询链路实现 `get_order_details` 工具。该工具底层对接真实 MySQL 订单、商品数据，支持订单号与数字订单 ID 双维度检索，并返回订单状态、支付状态、物流公司、物流单号、商品名称等结构化结果。

系统采用“两阶段生成”流程：第一阶段由 LLM 根据用户问题和上下文判断是否需要调用 `get_order_details`；第二阶段将工具从 MySQL 查询到的真实结果作为 `tool` 消息回填给模型，由模型基于事实数据生成最终客服回复。该机制将大模型定位为“决策与表达层”，将订单状态、物流信息等关键事实交给数据库和工具层提供，从根本上降低 AI 编造订单状态、物流进度或退款信息的风险。

### 业务级权限沙箱与 Guardrail 安全防线

在工具执行层设计基于买家、卖家、管理员的动态权限沙箱。订单查询并不是直接按用户输入查询数据库，而是先根据当前登录用户角色构造不同查询条件：买家只能查询自己的订单，卖家只能查询本店商品关联订单，管理员可以查询全量订单。该设计将权限控制下沉到工具执行层，即使 LLM 错误调用工具或用户恶意输入他人订单号，也无法越权获取敏感订单数据。

同时在 Agent 入口前置 Guardrail 校验，对银行卡、身份证、CVV、密码等敏感 PII 内容进行硬拦截，避免模型参与高风险隐私信息处理。系统还结合订单状态变更日志与运营审计能力，辅助追踪订单流转、退款处理和后台操作记录，使客服输出的数据不仅来自实时业务库，也具备可追溯的业务依据。

### 意图隔离与多轮上下文流转

设计面向电商客服场景的意图识别 Prompt，将用户输入分类为订单查询、政策规则咨询、商品咨询或闲聊等分支。对于连续追问场景，系统会读取最近历史消息，并将其组装进分类和生成 Prompt，使用户在上一轮提供订单号、商品信息后，下一轮可以继续追问物流、状态或规则问题。

在数据模型上，设计 `support_sessions` 与 `support_messages` 两层结构：`support_sessions` 管理客服会话，`support_messages` 按 `user / assistant / tool` 角色记录每轮输入、回复和工具结果。该设计既支撑多轮上下文恢复，也为后续问题复盘、客服质检、效果分析和知识库增强留下数据基础。

### LLM 高可用容灾与 Fallback 降级

统一封装定制化 `LLMClient`，屏蔽底层大模型接口差异，并集中处理模型超时、返回为空、`tool_calls` 参数格式漂移、JSON 解析异常、工具参数缺失等不稳定情况。业务层围绕不同场景配置多级 Fallback：意图识别失败时降级为闲聊或追问，订单工具调用失败时返回规则化提示，二阶段生成失败时直接基于工具结果拼装确定性回复。

该设计保证系统在 LLM 接口不可用、返回格式异常或工具调用链路部分失败时，仍能给出可控、合规、不会误导用户的回复。同时系统基于 Redis 对客服自动回复接口进行用户维度的短窗与长窗两级限流，并复用 Redis 限流能力保护登录、下单等高频业务接口，在异常流量或恶意请求场景下限制核心接口调用频率，降低 LLM Token 成本放大和后端数据库被打穿的风险。

## 实践成果

成功打通基于真实业务数据的 Agent 客服核心闭环，通过 MySQL 数据回填显著降低模型事实幻觉风险；同时结合角色级权限沙箱、敏感信息 Guardrail、Redis 两级限流与 `LLMClient` 多级 Fallback 容灾策略，有效兜底大模型调用异常、高频刷问、越权访问和敏感查询问题，沉淀出一套高可用、安全合规、可扩展的生产级智能客服后端架构。

从业务效果上，系统将订单查询、商品咨询、政策问答等高频客服问题从纯人工处理转为自动化处理，减少重复咨询的人力成本；从技术效果上，系统通过工具调用和真实数据回填提升回答准确性，通过权限沙箱和 Guardrail 提升数据安全性，通过 Fallback 和限流机制提升异常场景下的服务稳定性。

## 补充技术点说明

### FastAPI

使用 FastAPI 构建后端接口层，主要原因是其异步支持完善、类型提示友好，并能和 Pydantic、依赖注入、OpenAPI 文档自然集成。对于智能客服系统而言，LLM 调用、数据库访问和 Redis 访问都可能存在 IO 等待，FastAPI 的异步能力可以更好地承载并发请求，降低接口阻塞风险。

**解决的问题：** 统一承载客服会话、自动回复、订单、商品、退款、管理端等业务接口。

**带来的提升：** 接口结构清晰，便于扩展 Agent 能力；异步链路更适合 LLM 和数据库混合调用场景。

### SQLAlchemy Async + MySQL

使用 SQLAlchemy Async 访问 MySQL，将订单、商品、用户、客服会话、消息记录、退款和审计等业务实体持久化。MySQL 负责提供权威业务事实，Agent 不直接凭模型记忆回答订单状态，而是通过工具层查询数据库结果。

**解决的问题：** 解决大模型缺少实时业务状态、容易编造订单信息的问题。

**带来的提升：** 客服回复具备真实数据依据，订单状态、支付状态、物流信息等关键字段可追溯、可校验。

### Function Calling

使用 Function Calling 将 LLM 的自然语言理解能力和后端确定性业务能力连接起来。LLM 负责判断用户是否在问订单问题、是否具备订单号，工具层负责执行真实查询并返回结构化结果。

**解决的问题：** 避免让模型直接编造业务事实，同时避免所有问题都机械调用接口。

**带来的提升：** 在保持自然对话体验的同时，提高订单查询类回答的准确性和可控性。

### Guardrail

在 Agent 入口对敏感 PII 内容进行规则拦截，包括银行卡、身份证、CVV、密码等高风险信息。该层独立于 LLM，即使模型能力变化或 Prompt 失效，敏感内容仍会被前置拦截。

**解决的问题：** 降低隐私泄露、违规处理敏感信息、模型误答高风险问题的风险。

**带来的提升：** 提升客服系统合规性，使高风险输入不会进入后续模型生成链路。

### 角色级权限沙箱

在工具执行层按照用户角色构造查询范围，而不是只依赖接口层鉴权。买家、卖家、管理员分别对应不同的数据访问边界，订单工具最终只能返回当前用户有权查看的数据。

**解决的问题：** 防止用户通过构造订单号、诱导 Prompt 或绕过前端参数来查询他人订单。

**带来的提升：** 将安全边界固化在后端工具层，使 Agent 即使发生错误决策，也不会突破业务权限。

### Redis 限流

项目中使用 Redis 实现高频接口限流，对客服自动回复接口采用用户维度的短窗与长窗两级限流：短窗 10 秒最多 5 次，用于防止突发刷屏；长窗 1 分钟最多 20 次，用于防止持续恶意消耗 Token。同时登录、下单等关键业务接口也复用 Redis 限流能力进行频率控制。

**解决的问题：** 防止用户高频触发客服 Agent 导致 LLM Token 成本放大，也防止暴力登录、频繁下单或异常请求对核心业务接口造成冲击。

**带来的提升：** 提升系统抗压能力和故障隔离能力，避免高频请求击穿 LLM 调用链路、数据库或影响正常用户链路。

### LLMClient 统一封装

封装统一 `LLMClient` 管理模型请求、超时控制、返回解析、`tool_calls` 结构转换和异常抛出。业务层不直接依赖具体模型供应商返回结构，而是使用统一结果对象。

**解决的问题：** 降低模型接口不稳定、返回格式差异和后续模型替换带来的业务侵入。

**带来的提升：** 增强系统可维护性和可替换性，便于后续切换模型、调整参数或扩展更多工具。

### 多级 Fallback

围绕意图识别、工具调用、二阶段生成和普通对话生成分别设计兜底逻辑。例如意图分类失败时降级到闲聊，订单工具参数缺失时追问订单号，二阶段生成失败时使用工具结果拼装确定性回复。

**解决的问题：** 防止 LLM 超时、返回异常或生成失败时导致客服接口不可用。

**带来的提升：** 提升异常场景下的可用性，使核心客服链路即使在模型异常时仍能返回安全、确定性的结果。

### 会话持久化

使用 `support_sessions` 管理客服会话，使用 `support_messages` 记录多轮消息，并区分 `user / assistant / tool` 角色。工具结果也作为消息落库，形成完整的对话执行轨迹。

**解决的问题：** 解决连续追问缺少上下文、客服问题无法复盘、工具调用结果不可追踪的问题。

**带来的提升：** 支撑多轮对话、问题复盘、客服质检和后续 RAG/知识库优化。

### 审计日志与订单状态日志

项目中订单、退款等关键业务操作会写入状态日志或运营审计记录，管理端可以查看订单流转、退款处理和后台操作时间线。

**解决的问题：** 解决订单状态变化缺少追踪依据、客服或运营无法复盘业务处理过程的问题。

**带来的提升：** 提升业务可观测性和可追溯性，为客服回答、争议处理和后台管理提供依据。

### Pydantic 数据校验

使用 Pydantic 定义请求与响应 Schema，对客服消息、会话创建、自动回复请求等输入进行字段类型、长度和枚举校验。

**解决的问题：** 防止非法参数、超长内容或错误角色值直接进入业务逻辑。

**带来的提升：** 提升接口健壮性，减少运行时异常，也让前后端数据契约更清晰。

### JWT 鉴权与角色控制

系统通过登录认证获取当前用户，并在客服、订单、退款、管理端等业务接口中基于用户角色进行访问控制。

**解决的问题：** 区分买家、卖家、管理员不同操作权限，防止未授权访问。

**带来的提升：** 为 Agent 工具权限沙箱提供当前用户身份基础，使智能客服能力能够安全嵌入真实业务系统。

### 前端客服入口

前端已提供在线客服聊天入口，支持买家创建客服会话、加载历史消息、发送问题并展示自动回复。前端通过 API 调用后端 `support` 接口，形成基础的人机客服交互体验。

**解决的问题：** 后端 Agent 能力需要可交互入口承载，便于用户实际发起咨询和查看历史记录。

**带来的提升：** 打通从页面输入到后端 Agent 回复的用户链路，为后续继续优化客服体验、接入更多工具和知识库能力提供基础。
